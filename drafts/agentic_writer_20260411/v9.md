# Building the Agentic Writer: What I Learned Automating My Own Writing Pipeline
*Final | 2026-05-02 | ~1,460 words*

---

Building an agentic writing system taught me more about AI architecture than any framework comparison I've read. Here's what the system is, how it was built conversationally without code, and why the design principles apply to any knowledge workflow with distinct cognitive phases: proposal development, client briefings, board materials, vendor evaluation, competitive analysis. If that sequence sounds like your work, this article is for you regardless of whether you publish anything.

---

## What the System Is

The Agentic Writer is a set of 12 specialized agents running inside Claude Code, each with a defined role, defined inputs, and a defined output format.

The organizing principle is separation of concerns, the same design philosophy that makes complex software tractable. Content is separated from form. Logic is separated from polish. Scouting is separated from synthesis.

In practice, that means:

A **Knowledge Core** that maintains institutional memory across articles. An Archivist indexes everything I've written before, tracking my positions and stances. A Librarian surfaces relevant prior work before each new piece so I'm not inadvertently contradicting myself. A Stylist analyzes my writing patterns and maintains a voice profile: a plain-language document telling every other agent what my prose should sound like.

A **Production Team** that handles execution. An Interviewer conducts a structured intake session before research begins, surfacing the angle, audience, and intent. A Researcher scouts the web for current information, cross-referencing prior research and the knowledge core. A Ghostwriter drafts outlines and manuscripts from the research and POV brief. A Critic audits for logical fallacies, evidence gaps, and weak arguments. A Fact-Checker verifies claims, statistics, and quotes against sources before anything goes to final. An Editor applies final polish: brevity, mechanics, voice alignment. A Social Media Writer generates platform-specific posts for LinkedIn and X after the final draft is complete.

A **Persona System** that tests how different audiences receive a draft before it goes out. A Persona Generator creates realistic audience profiles from a short description: role, industry, key characteristics. A Persona Reviewer then embodies that profile and evaluates the draft from their perspective. The distinction matters: I'm not asking for generic audience feedback. I'm defining the specific skeptics and stakeholders relevant to a given topic, and the system reads the draft as them. This is the feature I underestimated most going in.

Each agent is scoped tightly. The Critic doesn't polish. The Editor doesn't research. The Ghostwriter doesn't fact-check. That's not inefficiency. It's the point. When an agent has a narrow, well-defined job, it does that job reliably.

## How It Was Actually Built

Here is the first message I sent: "Create agents that support a workflow like: research, draft, critique, edit."

That's it. No architecture document. No framework selection. No Python environment. I described a workflow the way I'd describe it to a new hire: roughly, directionally, with the expectation that we'd figure out the details together.

Then we iterated.

The Ghostwriter and Critic came first. Then I realized I needed the research to come in with structure, which led to the Researcher. Then I noticed I was redoing voice corrections in every draft, which led to the Stylist and the personal-voice skill. Then I noticed I was publishing things my target audience would have obvious objections to, objections I hadn't seen because I was too close to the argument, which led to the Persona system.

Each agent was built the same way: I described what it needed to do, what it should receive, and what it should produce. Claude generated the configuration. I tested it, pushed back, refined the scope. Most sessions ran under an hour, across two weeks of evening work, not a single sprint.

**The complexity didn't go away. The design tool changed.**

Every decision about how to scope an agent, how to sequence the handoffs, how to handle conflicts between what the research shows and what my prior stances claim, that was still my work. The thinking is still the human's job. What changed is that the interface is now language rather than code. I didn't bypass the architecture problem. I had a different conversation to solve it.

This works because of how Claude Code is structured. Every agent is defined as a markdown file: natural language instructions, not Python classes or graph nodes. Here is what a minimal agent definition looks like:

```
---
name: critic
description: Audits drafts for logical fallacies, evidence gaps, and weak arguments.
tools: read, write
---

You are the Critic. Your job is to audit the draft in [subfolder]/vN.md
and produce a redline report identifying: logical fallacies, unsupported claims,
evidence gaps, and weak arguments. You do not polish prose. You do not rewrite.
You identify structural problems and explain why they are problems.
Output your report to [subfolder]/critique.md.
```

Name, description, tools, and a system prompt in plain English. A user who can write a job description can write this. The prerequisite is structured thinking about what a role should and shouldn't do, which is a human skill, not a technical one.

Claude Code's configuration model is text-first in a way that LangGraph and CrewAI simply are not. LangGraph requires directed graph semantics. CrewAI requires a Python development environment as a baseline — though it introduced a visual enterprise tier in 2025, the code-first path remains the default for most users (sources 3, 4). Requiring a developer environment excludes most knowledge workers before they write a single line of configuration.

I used Claude Code because I'm comfortable on the command line. The underlying configuration model is the same whether you work in the terminal or a desktop environment. The agents are markdown files either way.

## What the System Still Cannot Do

The agents handle structural and research overhead. They maintain voice consistency. They surface blind spots. They catch logical gaps I'd miss on the fourth re-read of my own argument.

They do not handle judgment.

Deciding what's worth writing: mine. Selecting the angle that cuts through the noise: mine. Recognizing when new research contradicts a position I've held publicly for two years, and deciding what to do about that: definitely mine. The agents can flag the tension. They can't resolve it.

Agents produce drafts, not decisions. Output quality depends directly on how well the spec is written, and every handoff requires a human read.

The system also fails. Worth being honest about that. The most common failure mode: the Researcher returns a well-structured pack that confidently misses the most important recent development on a topic, and the Ghostwriter builds an argument on that incomplete foundation. I've caught this on re-read several times. The system doesn't self-correct here. The human review gate is the catch.

A second failure: agent scope drift. I've had the Critic start flagging style issues it has no business flagging, because a prompt tweak I made to solve one problem created ambiguity about its lane. The fix is always the same: tighten the spec. But it requires noticing the drift first. Configuration-based systems require the same maintenance discipline as code-based ones. That's not a reason to avoid them. It's a reason to version your agent files and treat prompt changes as real changes.

The persona system is the most useful tool for narrowing the judgment gap. Before finalizing one recent draft, I ran it through a persona built around my most skeptical expected reader: someone technically literate, deeply cynical about AI hype, predisposed to dismiss any "AI will change everything" framing. The review identified three soft spots I hadn't seen. Two made it into the revision.

That's not AI writing. That's AI as a pressure-testing tool, available on demand, without scheduling a call.

One more honest note: I am a technical professional. My comfort with a terminal is not a universal baseline. The argument isn't that anyone can build this tomorrow. It's that the barrier has moved substantially, and what remains is far more about clear thinking than technical fluency.

LangGraph, CrewAI, and AutoGen each require graph semantics, Python environments, and developer mental models as table stakes. Claude Code's conversational configuration path is a different on-ramp: not frictionless, but navigable without engineering skills. If 52% of professional developers don't yet use agents (Stack Overflow, December 2025), that figure is a floor, not a ceiling. The floor has moved. The ceiling hasn't disappeared.

## The Pattern Beyond Writing

That's what makes the pattern portable. It doesn't require pretending the agents handle judgment. It requires knowing they don't.

Writing is the example. Not the lesson.

The same cognitive sequence, gather information, synthesize against what you already know, draft an artifact, stress-test from an adversarial perspective, polish for the audience, applies to proposal development, client briefings, board materials, due diligence, competitive analysis. The workflow is the pattern. Writing is just where I happened to find it.

The persona system is especially portable. Don't use generic audience types. Build personas that match your actual clients, stakeholders, and skeptics: the CFO who only cares about payback period, the operations lead burned by the last three technology initiatives, the board member who asks the question everyone else avoids. Test your work against them before it leaves your desk.

Identify the cognitive phases in your workflow. Define what good output looks like at each stage. Scope each agent tightly to one phase. Iterate conversationally.

You don't need to know the architecture before you start. You need to know your workflow well enough to describe it, and be honest enough with the results to keep improving the spec.

---

## Start with One Phase

You do not need to wait.

Start with one phase of your workflow. Not the whole thing. Pick the phase where you spend the most time on structure and the least on judgment, the part that feels like scaffolding, not thinking. Describe what good output looks like. See what you can build.

This isn't about learning to code. It's about learning to specify. If you can describe your workflow clearly enough to explain it to a new hire, you can describe it clearly enough to build it with Claude.

The thinking is still yours. That was always the point.
