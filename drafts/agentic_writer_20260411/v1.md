# Building the Agentic Writer: What I Learned Automating My Own Writing Pipeline
*Draft v1 | 2026-04-11 | ~1,950 words*

---

Somewhere in the middle of building a multi-agent writing system with Claude, I realized the system itself was the more interesting story.

The original goal was practical: reduce the cognitive overhead of publishing. Every article I write involves the same structural overhead — gathering research, checking it against positions I've already staked out, drafting, arguing myself into corners, revising. Useful work. Also slow, repetitive work. Work that felt like it could be scaffolded.

So I built a pipeline. And then I noticed something. The pipeline didn't just handle the tedious parts. It changed how I thought about the work. And the way I built it — conversationally, iteratively, without writing a line of code — taught me more about agentic AI than any framework comparison I've read.

That's the thing worth sharing.

---

## What the System Is

The Agentic Writer is a set of 11 specialized agents running inside Claude Code, each with a defined role, a defined set of inputs, and a defined output format.

The organizing principle is separation of concerns — the same design philosophy that makes complex software tractable. Content is separated from form. Logic is separated from polish. Scouting is separated from synthesis.

In practice, that means:

A **Knowledge Core** that maintains institutional memory across articles. An Archivist indexes everything I've written before, tracking my positions and stances. A Librarian surfaces relevant prior work before each new piece so I'm not inadvertently contradicting myself. A Stylist analyzes my writing patterns and maintains a voice profile — a plain-language document that tells every other agent what my prose should sound like.

A **Production Team** that handles execution. An Interviewer conducts a structured intake session before research begins, surfacing the angle and audience and intent. A Researcher scouts the web for current information, cross-referencing both prior research and the knowledge core. A Ghostwriter drafts outlines and manuscripts from the research and POV brief. A Critic audits the draft for logical fallacies, evidence gaps, and weak arguments. An Editor applies final polish — brevity, mechanics, voice alignment.

A **Persona System** that tests how different audiences receive a draft before it goes out. I can define specific personas — not generic audience types, but the actual skeptics and stakeholders relevant to a given topic — and have the system evaluate the draft from their perspective. This is the feature I underestimated most going in. More on that shortly.

Each agent is scoped tightly. The Critic doesn't polish. The Editor doesn't research. The Ghostwriter doesn't fact-check. That's not inefficiency — it's the point. When an agent has a narrow, well-defined job, it does that job reliably.

## How It Was Actually Built

Here is the first message I sent: "Create agents that support a workflow like: research, draft, critique, edit."

That's it. No architecture document. No framework selection. No Python environment. I described a workflow the way I'd describe it to a new hire — roughly, directionally, with the expectation that we'd figure out the details together.

Then we iterated.

The Ghostwriter and Critic came first. Then I realized I needed the research to come in with structure, which led to the Researcher. Then I realized I was redoing voice corrections in every draft, which led to the Stylist and the personal-voice skill. Then I noticed I was publishing things my target audience would have obvious objections to — objections I hadn't seen because I was too close to the argument — which led to the Persona system.

Each agent was built in the same way: I described what it needed to do, what it should receive, and what it should produce. Claude generated the configuration. I tested it, pushed back on the results, refined the scope. Forty-five minutes per agent, on average, for the complex ones.

I want to be precise about what this means — because I've written skeptically about the "just vibe code it" model, and this could look like the same claim.

It's not.

**The complexity didn't go away. The design tool changed.**

Every decision about how to scope an agent, how to sequence the handoffs, how to handle conflicts between what the research shows and what my prior stances claim — that was still my work. The thinking is still the human's job. What changed is that the interface to build toward that thinking is now language rather than code. I didn't bypass the architecture problem. I had a different conversation to solve it.

## Why Claude Code Is Structurally Different

The dominant multi-agent frameworks — LangGraph, CrewAI, AutoGen — all carry a developer prerequisite. Not as an incidental barrier. As an architectural fact.

LangGraph requires understanding directed graph semantics — you are literally constructing a state machine, defining nodes and edges and how information propagates across them. AutoGen requires advanced Python. CrewAI is described, charitably, as "the most accessible" of the three. It still requires Python 3.10 or above and a developer workflow to do anything beyond a tutorial. As a competitor comparison noted directly: "Non-technical teams: If your team doesn't have Python developers, MindStudio removes technical barriers completely." The framing is telling. The assumed baseline is that CrewAI requires Python developers. That's not a UX limitation. It's an architectural one.

Claude Code is different, and not by accident. The tool began as an AI-assisted coding tool — its configuration model inherited the conversational, text-first paradigm of AI interaction rather than the schema-first paradigm of software frameworks.

The primitive unit of configuration is a markdown file written in natural language. CLAUDE.md is a plain-text document that provides persistent context across every session. Sub-agents are markdown files: a name, a description, a list of available tools, and a plain-language system prompt. Skills — like the voice profile my Stylist maintains — are SKILL.md files, written as instruction documents. If you can write a job description, you can write a sub-agent definition.

There is now a community repository of 100+ pre-built sub-agents available on GitHub. That level of community contribution happens when the primitive is accessible enough to be shared — when defining an agent is a writing task, not an engineering task.

I want to be honest about where the ceiling is. Claude Code is a command-line tool. Getting it installed and running requires some comfort with a terminal. That is a real bar — meaningfully lower than building with LangGraph, meaningfully higher than clicking through a Zapier template. The FreeCodeCamp Claude Code Handbook put it well: "The ability to build software is becoming accessible to a broader range of people — not because the underlying complexity has been eliminated, but because much of the mechanical translation between intent and implementation can now be delegated to an AI agent."

That's the right framing. The complexity is still there. The interface to it changed.

## What the System Still Cannot Do

The agents handle the structural and research overhead. They maintain voice consistency. They surface blind spots. They catch logical gaps I'd miss on the fourth re-read of my own argument.

They do not handle judgment.

Deciding what's worth writing — that's mine. Selecting the angle that cuts through the noise on a given topic — mine. Recognizing when new research contradicts a position I've held publicly for two years, and deciding what to do about that — definitely mine. The agents can flag the tension. They can't resolve it.

The persona system is the most useful tool for narrowing the judgment gap. Before I finalized one recent draft, I ran it through a persona built around the most skeptical reader I expected to encounter: someone technically literate, deeply cynical about AI hype, and predisposed to dismiss any "AI will change everything" framing. The persona review identified three places where my argument had soft spots I hadn't seen. Two of them made it into the revision.

That's not AI writing. That's AI as a pressure-testing tool — available on demand, without requiring me to schedule a call.

I should also say this plainly: I am a technical professional. My comfort with a terminal is not a universal baseline. The honest argument is not that anyone can build this tomorrow. It's that the barrier has moved substantially — and that the remaining barrier is much more about clear thinking than it is about coding skill.

According to the Stack Overflow 2025 Developer Survey, 52% of professional developers either don't use agents or stick to simpler AI tools. If the adoption gap is that wide even among technical users, the gap for non-technical practitioners is substantial. But it's not infinite. And it's closing.

## The Pattern Beyond Writing

Writing is the example. Not the lesson.

Any repetitive knowledge workflow with distinct cognitive phases has the same structural opportunity. Consider what the phases actually are: gather information, synthesize it against what you already know, draft an artifact, stress-test it from an adversarial perspective, polish it for the intended audience. That sequence — research, synthesize, draft, critique, polish — applies to proposal development, client briefings, board materials, due diligence, competitive analysis. The workflow is the pattern. Writing is just where I happened to find it.

The persona system is especially portable. You don't need generic audience types. Build personas that match your actual clients, stakeholders, and skeptics — the CFO who only cares about payback period, the operations lead who's been burned by the last three technology initiatives, the board member who asks the question everyone else is afraid to ask. Test your work against them before it leaves your desk.

The organizing principle is always the same:

Identify the cognitive phases in your workflow. Define what good output looks like at each stage. Scope each agent tightly to one phase. Iterate toward it conversationally.

That last part is the new thing. You don't need to know the architecture before you start. You need to know your workflow well enough to describe it — and then you need to be honest enough with the results to keep improving the spec.

---

## Start with One Phase

The AI agent market was $7.63 billion in 2025 and is projected to reach $182.97 billion by 2033 — a 49.6% compound annual growth rate, according to Grand View Research. The tooling will mature. The use cases will multiply. Visual interfaces will lower the floor further.

You do not need to wait.

Start with one phase of your workflow. Not the whole thing. Pick the phase where you spend the most time on structure and the least on judgment — the part that feels like scaffolding, not thinking. Describe what good output looks like. See what you can build.

This isn't about learning to code.

It's about learning to specify. And if you can describe your workflow clearly enough to explain it to a new hire, you can describe it clearly enough to build it with Claude.

The thinking is still yours. That was always the point.
