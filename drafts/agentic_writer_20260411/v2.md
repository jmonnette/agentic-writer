# Building the Agentic Writer: What I Learned Automating My Own Writing Pipeline
*Draft v2 | Post-Critique Revision | 2026-04-22 | ~1,610 words*

---

Somewhere in the middle of building a multi-agent writing system with Claude, I realized the system itself was the more interesting story.

The original goal was practical: reduce the cognitive overhead of publishing. Every article I write involves the same structural overhead — gathering research, checking it against positions I've already staked out, drafting, revising. Useful work. Also slow, repetitive work that felt like it could be scaffolded.

So I built a pipeline. And then I noticed something. The pipeline didn't just handle the tedious parts. It changed how I thought about the work. And the way I built it — conversationally, iteratively, without writing a line of code — taught me more about agentic AI than any framework comparison I've read.

That's the thing worth sharing.

---

## What the System Is

The Agentic Writer is a set of 11 specialized agents running inside Claude Code, each with a defined role, defined inputs, and a defined output format.

The organizing principle is separation of concerns — the same design philosophy that makes complex software tractable. Content is separated from form. Logic is separated from polish. Scouting is separated from synthesis.

In practice, that means:

A **Knowledge Core** that maintains institutional memory across articles. An Archivist indexes everything I've written before, tracking my positions and stances. A Librarian surfaces relevant prior work before each new piece so I'm not inadvertently contradicting myself. A Stylist analyzes my writing patterns and maintains a voice profile — a plain-language document that tells every other agent what my prose should sound like.

A **Production Team** that handles execution. An Interviewer conducts a structured intake session before research begins, surfacing the angle, audience, and intent. A Researcher scouts the web for current information, cross-referencing both prior research and the knowledge core. A Ghostwriter drafts outlines and manuscripts from the research and POV brief. A Critic audits the draft for logical fallacies, evidence gaps, and weak arguments. An Editor applies final polish: brevity, mechanics, voice alignment.

A **Persona System** that tests how different audiences receive a draft before it goes out. I can define specific personas — not generic audience types, but the actual skeptics and stakeholders relevant to a given topic — and have the system evaluate the draft from their perspective. This is the feature I underestimated most going in.

Each agent is scoped tightly. The Critic doesn't polish. The Editor doesn't research. The Ghostwriter doesn't fact-check. That's not inefficiency. It's the point. When an agent has a narrow, well-defined job, it does that job reliably.

## How It Was Actually Built

Here is the first message I sent: "Create agents that support a workflow like: research, draft, critique, edit."

That's it. No architecture document. No framework selection. No Python environment. I described a workflow the way I'd describe it to a new hire — roughly, directionally, with the expectation that we'd figure out the details together.

Then we iterated.

The Ghostwriter and Critic came first. Then I realized I needed the research to come in with structure, which led to the Researcher. Then I noticed I was redoing voice corrections in every draft, which led to the Stylist and the personal-voice skill. Then I noticed I was publishing things my target audience would have obvious objections to — objections I hadn't seen because I was too close to the argument — which led to the Persona system.

Each agent was built the same way: I described what it needed to do, what it should receive, and what it should produce. Claude generated the configuration. I tested it, pushed back, refined the scope. For the complex agents, most sessions ran under an hour — across two weeks of evening work, not a single sprint.

**The complexity didn't go away. The design tool changed.**

Every decision about how to scope an agent, how to sequence the handoffs, how to handle conflicts between what the research shows and what my prior stances claim — that was still my work. The thinking is still the human's job. What changed is that the interface to build toward that thinking is now language rather than code. I didn't bypass the architecture problem. I had a different conversation to solve it.

This works because of how Claude Code is structured. Every agent is defined as a markdown file — natural language instructions, not Python classes or graph nodes. Claude Code's configuration model is text-first in a way that LangGraph (which requires directed graph semantics) and CrewAI (which requires Python 3.10+ as baseline) simply are not. A user who can write a job description can write an agent definition. That's not a marketing claim — it's an architectural fact with real consequences for who can build this without help.

I used Claude Code because I'm comfortable on the command line. If you prefer a desktop interface, Claude Cowork provides the same agentic capabilities through a visual environment built for knowledge workers — the underlying primitives are identical, the terminal is just gone.

## What the System Still Cannot Do

The agents handle structural and research overhead. They maintain voice consistency. They surface blind spots. They catch logical gaps I'd miss on the fourth re-read of my own argument.

They do not handle judgment.

Deciding what's worth writing — mine. Selecting the angle that cuts through the noise on a given topic — mine. Recognizing when new research contradicts a position I've held publicly for two years, and deciding what to do about that — definitely mine. The agents can flag the tension. They can't resolve it.

One practical note that's easy to skip over: agents produce drafts, not decisions. Output quality depends directly on how well the spec is written, and every handoff requires a human read. The system catches most of what I'd miss; it doesn't eliminate the verification step.

The persona system is the most useful tool for narrowing the judgment gap. Before finalizing one recent draft, I ran it through a persona built around the most skeptical reader I expected: someone technically literate, deeply cynical about AI hype, predisposed to dismiss any "AI will change everything" framing. The review identified three soft spots I hadn't seen. Two made it into the revision.

That's not AI writing. That's AI as a pressure-testing tool — available on demand, without scheduling a call.

One more honest note: I am a technical professional. My comfort with a terminal is not a universal baseline. The argument isn't that anyone can build this tomorrow. It's that the barrier has moved substantially — and the remaining barrier is far more about clear thinking than technical fluency.

The dominant multi-agent frameworks — LangGraph, CrewAI, AutoGen — require graph semantics, Python environments, and developer mental models as table stakes. That barrier keeps most knowledge workers out entirely. Claude Code's conversational configuration path is a different on-ramp: not frictionless, but genuinely navigable without engineering skills. If even 52% of professional developers don't yet use agents (Stack Overflow, December 2025), the gap for non-technical practitioners is real — but the reason isn't that the tools are equally inaccessible. The floor has moved. The ceiling hasn't disappeared.

## The Pattern Beyond Writing

Writing is the example. Not the lesson.

Most repetitive knowledge workflows with distinct cognitive phases have the same structural opportunity. The phases: gather information, synthesize it against what you already know, draft an artifact, stress-test it from an adversarial perspective, polish it for the intended audience. That sequence — research, synthesize, draft, critique, polish — applies to proposal development, client briefings, board materials, due diligence, competitive analysis. The workflow is the pattern. Writing is just where I happened to find it.

The persona system is especially portable. Don't use generic audience types. Build personas that match your actual clients, stakeholders, and skeptics: the CFO who only cares about payback period, the operations lead burned by the last three technology initiatives, the board member who asks the question everyone else avoids. Test your work against them before it leaves your desk.

The organizing principle is always the same: identify the cognitive phases in your workflow, define what good output looks like at each stage, scope each agent tightly to one phase, and iterate conversationally.

That last part is the new thing. You don't need to know the architecture before you start. You need to know your workflow well enough to describe it — and be honest enough with the results to keep improving the spec.

---

## Start with One Phase

The AI agent market was $7.63 billion in 2025 and is projected to reach $182.97 billion by 2033 (Grand View Research). The window to be an early mover in your domain — before this becomes table stakes in your industry — is still open. That projection is evidence the window is real, not just that the technology is large.

You do not need to wait.

Start with one phase of your workflow. Not the whole thing. Pick the phase where you spend the most time on structure and the least on judgment — the part that feels like scaffolding, not thinking. Describe what good output looks like. See what you can build.

This isn't about learning to code. It's about learning to specify. If you can describe your workflow clearly enough to explain it to a new hire, you can describe it clearly enough to build it with Claude.

The thinking is still yours. That was always the point.
