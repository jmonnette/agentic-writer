# The Token Bill Is Coming. Is Your Team Ready?
*Draft v1 | 2026-04-26 | ~2,050 words*

On June 1, GitHub Copilot stops being "unlimited." Not for everyone, and not all at once — but the era of flat-rate frontier model access is ending. Under the new AI Credits model, plans retain their current price points while converting that spend into usage credits. Overages are billed at consumption rates. Teams that never thought about which model they were using will start noticing when the credits run out mid-sprint.

This is the forcing function. Finance is already watching. CFOs scrutinize AI spend with the same lens they apply to cloud infrastructure — and unmanaged cost centers get shut down. The question is not whether you will be asked to justify what you are spending on AI. The question is whether you have built the habits to answer that question.

Before going further: one clarification that matters. I have written before that tracking tokens as a performance metric is a form of metric theater — like measuring agile by counting stand-ups. That position stands. Token spend is not a KPI to report upward. It is a cost input, managed internally by practitioners the same way cloud infrastructure spend is managed. Nobody lets AWS bills run unsupervised; nobody should let AI API bills run that way either. Managing it does not mean measuring it as a proxy for engineering productivity. It means treating it like the operational cost it is.

## The Pricing Landscape Has Fundamentally Changed

Three phases in 24 months. In 2024, most AI coding tools offered flat subscriptions — Cursor's "unlimited fast requests" model was the template. By mid-2025, as frontier model costs rose and heavy users strained infrastructure, tools introduced credit pools. Cursor switched first (June 2025), announcing it poorly. By 2026, the industry is converging on usage-based billing. GitHub Copilot's June shift is the clearest signal yet that this direction is settled.

The major tools now break down like this:

**Claude Code** offers tiered subscriptions ($20 Pro / $100 Max 5x / $200 Max 20x) with rolling 5-hour usage windows rather than monthly ceilings. API billing runs $1/$5 per million tokens for Haiku 4.5, $3/$15 for Sonnet 4.6, and $5/$25 for Opus 4.7 (input/output respectively). The gap between subscription and API rates is striking: one developer tracked eight months of daily usage across 10 billion tokens — over $15,000 at API rates, approximately $800 on the Max plan. A 93% difference. That is not an edge case; it is what happens when teams default to heavy API-billed usage without evaluating alternatives.

**GitHub Copilot** runs $10/month Pro, $19/user Business, $39/user Enterprise. Included models (GPT-5 mini, GPT-4.1, GPT-4o) consume no premium requests. Claude Sonnet costs 0.9x multiplier on auto-select; Claude Opus consumes significantly more. The June 1 transition converts plan value to AI Credits — Business teams get a promotional $30/month credit buffer through August, Enterprise gets $70/month. After that, the real consumption picture becomes visible.

**Cursor** charges $20/month Pro with a $20 monthly credit pool for manual frontier model selection. Auto mode is unlimited and draws from a rotating pool — the recommended default for routine work. Manual frontier model selection depletes credits at model-specific rates.

**Windsurf** overhauled pricing in March 2026, raising Pro from $15 to $20/month and adding a $200/month Max tier. Tab autocomplete remains unlimited; Cascade (multi-file agent) and Chat with premium models consume quota. A three-step refactoring flow in Claude Sonnet 4.5 costs roughly twice the credits of the same flow in Kimi K2.

The strategic pattern across all of these tools is identical: a base model tier that is free or effectively unlimited, and a frontier model tier that costs real money per use. Teams that understand this structure can stay inside the unlimited tier for the majority of their work. Teams that do not understand it spend frontier model credits on tasks that did not require them.

## The Most Common Failure: Defaulting to Opus for Everything

The single most expensive mistake engineering teams make is treating every AI task as if it requires the most capable model available. It does not. Most AI coding work does not.

Code autocomplete. Tab completion. Single-file edits. Documentation generation. Rote test generation. All of these are pattern recognition tasks. They do not require the extended reasoning capacity of a frontier model. Running them through Claude Opus 4.7 at $25 per million output tokens when Claude Haiku 4.5 at $5 per million would deliver equivalent output is not just wasteful — it is the same failure mode as commissioning custom engineering on a problem that a SaaS module already solves. The author's prior work on enterprise build-vs-buy decisions puts it plainly: don't spend custom engineering on solved problems. The same logic applies to model selection.

A practical task-to-model mapping based on practitioner consensus across tools:

- **Fast/included model** (Haiku, GPT-4o mini, Auto mode): Autocomplete, tab completion, single-file edits, documentation, rote test generation, formatting
- **Mid-tier** (Sonnet, GPT-4.1): Multi-file refactoring, debugging with a stack trace, codebase onboarding, code explanation
- **Frontier** (Opus, o3): Architectural planning, complex algorithm design, multi-system reasoning, security audits

Most work falls in the first two categories. Frontier models should be reserved for tasks where their extended reasoning capacity actually changes the output — not used by default because they are the "best" model.

Effort level matters too, specifically in Claude Code. The `/effort` parameter controls how much internal reasoning the model performs before responding. At lower effort, Claude still thinks on sufficiently complex problems — it just thinks less. Higher effort means more internal reasoning tokens and higher cost.

In March 2026, Anthropic reduced Claude Code's default effort from High to Medium to cut latency and limit usage. Users reported noticeable quality degradation over several weeks. Anthropic reversed the change on April 7, 2026, and published a postmortem on April 23: "This was the wrong tradeoff. We reverted this change after users told us they'd prefer to default to higher intelligence and opt into lower effort for simple tasks."

The incident teaches two things. First, effort level has a measurable quality cost when lowered indiscriminately. Second, the right approach is practitioner-controlled: lower effort for simple tasks, maintain high effort for complex ones. Setting `MAX_THINKING_TOKENS=8000` as a ceiling for routine tasks is standard cost-control practice without touching the default for complex work.

## Context Engineering: The Highest-Leverage Intervention

Model selection and effort tuning address what you spend per operation. Context engineering addresses how much context each operation requires. For most teams, context is the larger problem.

A 50-tool MCP agent wastes up to 13,000 tokens on the tool catalog before processing any message. Three connected MCP services — a realistic enterprise configuration — can burn 143,000 tokens of a 200,000 token window before the agent begins any actual work. That is 72% of the available context spent before the first line of code is generated. The CNCF documented a case where a single "summarize meeting notes" request with 114 tools injected consumed 240,600 tokens total, the vast majority from schema definitions the model never needed.

This is MCP tool bloat. It is the fastest-growing token cost vector in agentic AI workflows and the least visible one, because the cost is paid silently before productive work begins.

Context hygiene has several practical layers:

**CLAUDE.md discipline**: 72.6% of Claude Code projects specify application architecture in configuration files (ArXiv, n=2,303). These files load at session start and provide context that survives restarts. But every token in CLAUDE.md is a token unavailable for conversation. The guidance is to keep CLAUDE.md under 500 tokens: document what the AI gets wrong, not comprehensive manuals. Reference file paths rather than embedding entire documents.

**MCP tool auditing**: Claude Code's `/context` command shows the actual token breakdown in your current session. Start there. If you have more than 20 active MCP tools, apply schema compression. Atlassian's open-source `mcp-compressor` reduces the Atlassian MCP server's schema overhead by 70–97% by collapsing dozens of individual tools into wrapper functions that preserve the interface but strip verbose descriptions. For catalogs above 50 tools, dynamic tool loading — exposing only semantically relevant tools per request rather than the full catalog — reduces context overhead by 96x or more, with documented 100% task success rates in Speakeasy's benchmarks.

**Session management**: The recommended discipline is Explore, Decide, Summarize, then `/compact` before continuing. Converting recurring context to one-time context is the core principle. `/clear` wipes context entirely — appropriate when switching to an unrelated task, not mid-session.

**Response compression**: Tool output flooding, where agents pipe verbose command responses directly into context, is one of the top three token waste vectors. RTK (Rust Token Killer) is a CLI proxy that intercepts shell command outputs and compresses them before they reach the model's context. Practitioners report 60–90% reductions on CLI-heavy workflows; one user saved 10 million tokens (89%) across Claude Code sessions. The caveat: RTK can interfere with complex multi-turn agentic sessions. It is most reliable for single-command, read-heavy operations.

**The key insight from practitioners**: a model on Low effort with great context often outperforms the same model on Max effort with poor context. The extra thinking budget gets spent reconstructing state the session should already have. Context quality and model tier are not independent levers. They interact. Address both.

## Local Models and Self-Hosting: A Real Option for the Right Team

This is worth taking seriously. Not for every team, and not as a cost-cutting reflex — but as a genuine strategic option when two conditions are met: high token volume and proprietary codebase requirements.

Open-source coding models in 2026 are genuinely competitive with frontier models for many coding tasks. Qwen2.5-Coder-32B scores 37.2% on LiveCodeBench, compared to GPT-4o's 29.2%. The 7B variant runs on a gaming laptop at 28–35 tokens per second. The 32B requires roughly 18GB VRAM at Q4 quantization. The performance gap versus frontier models narrows considerably for code generation, completion, and repair tasks.

The break-even economics are volume-dependent. Conservative estimates put the threshold at around 11 billion tokens per month before self-hosting a 7B model on an H100 ($0.013/1,000 tokens) beats API pricing. More aggressive estimates from premium model comparison put the threshold at 5–10 million tokens per month. A Mac Mini M4 Pro (48GB, approximately $2,000) can run 30B coding models and serve a small team. A Mac Studio M4 Ultra (192GB, approximately $8,000–$10,000) runs quantized 70B models at 2–5 tokens per second — workable for batch tasks, slow for interactive use.

For most individual developers, the Max plan subscription is simpler, better-supported, and economically rational. The self-hosting case becomes compelling for teams processing very high token volumes, particularly if those teams have proprietary codebases they cannot send to cloud APIs.

Which brings in fine-tuning. Fine-tuning a smaller open-source model on proprietary code can yield significant accuracy gains for specialized tasks. Together AI published results showing 60% better accuracy and 10–100x lower inference cost for specialized tasks using fine-tuned small models versus large closed-source models. The counterpoint is real: the practitioner community is largely skeptical for codebases under 1 million tokens, citing insufficient training data, drift as code evolves, and the ongoing maintenance burden of retraining. Fine-tuning makes sense when the task is highly repetitive and well-defined, the codebase is large and stable, volume is high enough to amortize training cost, and privacy requirements prevent cloud API use.

Absent those conditions, RAG over a well-maintained CLAUDE.md or `.cursorrules` file typically delivers comparable domain-specificity gains at a fraction of the operational cost. It is not as powerful. But it requires no retraining when your codebase changes.

The two strategies — self-hosting and fine-tuning — are linked, not independent options. Self-hosting earns its operational cost most clearly when fine-tuning is in the picture. Without a fine-tuning rationale, you are usually better served by Haiku-class cloud models with aggressive caching.

## What to Actually Do: Building Team Practices

Awareness is not enough. Here is what practitioners who have navigated this establish for their teams.

**Audit before optimizing.** Run Claude Code's `/context` command. Look at your MCP tool count. Identify your highest-volume use cases and which model tier they are currently using. The picture is usually worse than expected.

**Establish model tier defaults by task type.** Do not leave model selection to individual developer judgment. Define a team-level policy: Auto or fast model for autocomplete and single-file edits, mid-tier for multi-file refactoring and debugging, frontier only for architecture and complex reasoning. Review periodically.

**Treat your CLAUDE.md as a cost asset.** Every token that loads at session start loads on every session. Keep it focused. What does the AI consistently get wrong in your codebase? Put that in. Comprehensive documentation belongs elsewhere.

**Audit your MCP footprint.** If you are running more than 20 active MCP tools, you have schema bloat. Use `/context` to measure it. Apply `mcp-compressor` to high-tool servers. Consider whether you need every connected server active for every session.

**Choose subscription plans over API billing for predictable team work.** The $15,000 vs. $800 comparison is an extreme case. It illustrates the structural point: subscriptions offer a cost ceiling that API billing does not. Reserve API-direct billing for automation pipelines where caching and batching can reduce effective rates by 50–90%.

**For Claude Code users, calibrate effort level explicitly.** High effort is the right default for complex tasks. Set `MAX_THINKING_TOKENS=8000` for routine work where you do not need full extended reasoning. Do not lower the default globally — the Anthropic incident showed what happens when that tradeoff is made at platform level rather than task level.

## The Stakes Are Not Abstract

AI tool costs are doubling year over year. Global LLM API spend reached $8.4 billion in 2025. Finance teams are paying attention. CFOs who evaluate AI ROI through cost savings, risk reduction, and revenue growth need programs that can demonstrate cost discipline — not just output volume.

Teams that build these practices now are not optimizing for its own sake. They are ensuring their AI programs remain funded, visible, and expanding. The ones that don't are running unmanaged cost centers. Unmanaged cost centers get shut down.

Failing to develop tokenomics skills does not put you in the same category as teams that reject AI entirely. But it puts you closer than you want to be. Efficiency is becoming the baseline, not the differentiator.

The hard, unglamorous work of making AI usage sustainable — auditing your tool footprint, calibrating your model tier defaults, enforcing CLAUDE.md discipline — is exactly where the value gets protected. Not just created. Protected.

Start with the audit.
